---
title: "Rate limiting"
description: "How rate limiting works in Bitácora Operativa and how to configure the limits."
---

Bitácora Operativa ships with two independent rate limiters: one for the general API and one for the login endpoint. Both are disabled by default in local environments and enabled in production.

## Limiters

### API limiter

Applied to all `GET /api/v1/*` routes.

| Variable | Default (production) | Default (local) | Description |
|---|---|---|---|
| `RATE_LIMIT_API_ENABLED` | `true` | `false` | Enable or disable the limiter. |
| `RATE_LIMIT_API_MAX_ATTEMPTS` | `60` | `6000` | Requests allowed per window. |
| `RATE_LIMIT_API_DECAY_MINUTES` | `1` | `1` | Window length in minutes. |

### Login limiter

Applied to `POST /login`.

| Variable | Default (production) | Default (local) | Description |
|---|---|---|---|
| `RATE_LIMIT_LOGIN_ENABLED` | `true` | `false` | Enable or disable the limiter. |
| `RATE_LIMIT_LOGIN_MAX_ATTEMPTS` | `5` | `50` | Attempts allowed per window. |
| `RATE_LIMIT_LOGIN_DECAY_SECONDS` | `60` | `60` | Window length in seconds. |

## Configuring limits

Set the environment variables in your `.env` file:

```bash
RATE_LIMIT_API_ENABLED=true
RATE_LIMIT_API_MAX_ATTEMPTS=120
RATE_LIMIT_API_DECAY_MINUTES=1

RATE_LIMIT_LOGIN_ENABLED=true
RATE_LIMIT_LOGIN_MAX_ATTEMPTS=5
RATE_LIMIT_LOGIN_DECAY_SECONDS=60
```

After changing any rate limit variable, clear the configuration cache:

```bash
php artisan optimize:clear
```

<Warning>
  Rate limit configuration is cached by Laravel. Changes to `.env` have no effect until you run `php artisan optimize:clear`.
</Warning>

## HTTP 429 response

When a client exceeds the limit, the server responds with `HTTP 429 Too Many Requests`.

```json
{
  "message": "Too Many Attempts."
}
```

The response includes a `Retry-After` header indicating how many seconds to wait before retrying:

```
HTTP/1.1 429 Too Many Requests
Retry-After: 42
X-RateLimit-Limit: 60
X-RateLimit-Remaining: 0
```
